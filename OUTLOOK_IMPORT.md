# Outlook Email → Static Template Import

Design + phased implementation plan for importing an Outlook email (`.msg` / `.eml` / pasted HTML) and auto-generating an editable static template composed of sections from the existing **Section Library**.

The output feeds directly into the current template pipeline, so preview HTML and Outlook-optimized HTML are regenerated by the existing generators — no divergence.

---

## 1. Goal

Today, static templates are created by drag-and-drop in `src/pages/TemplateEditor.tsx`. Sections are stored as `Section[]` (see `src/types/section.ts`) and rendered by `src/lib/templateUtils.ts` into preview HTML and by the Outlook generator into MSO-safe HTML.

**New requirement:** allow the user to upload an existing Outlook email. The system parses it, maps each block to a `SectionType` from the section library, and reconstructs an editable `Template` — visually and structurally equivalent to a manually built one.

---

## 2. High-Level Flow

```text
Upload (.msg | .eml | .html | pasted HTML)
        │
        ▼
[1] Extract raw HTML + inline assets (images, cid: refs)
        │
        ▼
[2] Normalize Outlook/MSO HTML → clean semantic DOM (jsoup)
        │
        ▼
[3] Segment DOM into logical blocks (walk top-level nodes)
        │
        ▼
[4] Classify each block → SectionType (rule-based, extensible)
        │
        ▼
[5] Build Section JSON (content, styles, variables) per block
        │
        ▼
[6] Detect placeholders → auto-create {{variables}}
        │
        ▼
[7] Assemble Template → open in TemplateEditor (editable, previewable)
        │
        ▼
[8] Show Import Report (mapped / fallback / warnings)
```

## 4. HTML Normalization Rules

Reverse of the existing Outlook HTML generator:

- Strip `<!--[if mso]>…<![endif]-->` and `<!--[if !mso]>…<![endif]-->`.
- Unwrap `<table role="presentation">` used purely for spacing (heuristic: single cell, no borders, no `<th>`, single content child).
- **Keep** semantic tables — heuristic: has `<th>`, OR (>1 row AND >1 col AND text-heavy cells).
- Preserve inline `style="…"` — the `Section.styles` object supports these directly.
- Rewrite `<img src="cid:xxx">` → hosted URL after asset upload.
- Collapse `&nbsp;` runs; normalize whitespace but preserve intentional `<br>`.
- Normalize fonts to `'Wells Fargo Sans'` where the source declared a matching family, else leave the source font as-is.

## 5. Block Classifier — Priority Table

First match wins. Each classifier implements `boolean matches(Element el)` + `int priority()` + `Section build(Element el)`.

| Priority | Classifier              | Matches (heuristic)                                                             | → `SectionType`                    |
| -------- | ----------------------- | ------------------------------------------------------------------------------- | ---------------------------------- |
| 100      | `BannerClassifier`      | First block, full-width, colored bg, contains hero text/image                   | `banner`                           |
| 90       | `ProgramNameClassifier` | Prominent centered text near top matching a known program-name pattern          | `program-name`                     |
| 80       | `HeadingClassifier`     | `<h1>`–`<h6>`                                                                   | `heading1`…`heading6`              |
| 75       | `GifClassifier`         | `<img>` with `.gif` extension or `image/gif` MIME                               | `gif`                              |
| 70       | `ImageClassifier`       | Standalone `<img>` in its own block                                             | `image`                            |
| 65       | `ButtonClassifier`      | `<a>` styled as button (bg-color + padding + border-radius) OR class `.btn`     | `button`                           |
| 60       | `TableClassifier`       | Semantic `<table>` (has `<th>` or clearly tabular)                              | `table`                            |
| 55       | `ListClassifier`        | `<ul>`/`<ol>` — sub-type by `list-style-type` / `type` attribute                | `bullet-list-*` / `number-list-*`  |
| 50       | `SeparatorClassifier`   | `<hr>` or single-row table used as a rule                                       | `separator-line`                   |
| 45       | `LineBreakClassifier`   | Empty `<p>`, `<br>`-only block, or spacer row                                   | `line-break`                       |
| 40       | `LabeledContentClassifier` | `label: value` pattern (`<strong>Label:</strong> value`)                     | `labeled-content`                  |
| 35       | `MixedContentClassifier`   | Paragraph with mixed inline `<a>` + text and multiple placeholders           | `mixed-content`                    |
| 30       | `FooterClassifier`      | Last block, small font, disclaimer/contact keywords                             | `footer`                           |
| 20       | `ParagraphClassifier`   | Plain `<p>` or `<div>` of text                                                  | `paragraph`                        |
| 0        | `HtmlContentFallback`   | Anything else — kept as raw HTML, editable                                      | `html-content`                     |

Add new rules by dropping a new `BlockClassifier` bean; ordering by `priority()` keeps behavior predictable.

## 6. Section Construction

For each classified block, build a `Section` (matches `src/types/section.ts`):

```ts
{
  id: uuid(),
  type: <classifiedType>,
  content: element.text() | inlineHtml,
  styles: extractInlineStyles(element),   // fontSize, color, bg, align, weight…
  variables: {},                          // populated by placeholder detector
  order: index,
}
```

- **Tables** → build `variables.tableData` matching the shape consumed by `TableEditor.tsx`.
- **Lists** → build `ListItemStyle[]` respecting up to 3 levels of nesting (per memory `nested-multi-level-lists`).
- **Mixed content** → tokenize inline `<a>` anchors into placeholder slots the editor understands.
- **Banner / program-name** → single-use rule enforced (memory `single-use-sections`); if source contains 2+, only the first becomes the section, the rest fall back to `html-content`.

---

## 7. Placeholder Detection

- Normalize existing patterns to `{{var}}`: `[NAME]`, `<%= foo %>`, `${bar}`, `{{baz}}`, `%%VAR%%` all → `{{var}}`.
- Extract via existing `src/lib/variableExtractor.ts` after normalization so the template's variable panel is auto-populated.
- **Optional v2 heuristic:** literal dates (`Nov 30, 2025`), currency (`$1,200.00`), URLs → suggest a variable with the literal value as the default.

---

## 8. Import Report

Shown after import completes:

- ✅ N sections mapped (`heading`, `table`, `image`, …)
- ⚠️ M blocks fell back to `html-content` (click to jump to that section in the editor)
- 🖼️ K images uploaded
- 🔤 P placeholders detected

Backed by an `ImportReport` object in `ImportedTemplateDto.report` with per-block entries `{ index, chosenType, confidence, notes }`.

## 9. Phased Rollout

| Phase | Scope                                                                              | Deliverable file                    |
| ----- | ---------------------------------------------------------------------------------- | ----------------------------------- |
| 1     | MVP — paste HTML, basic classifiers (headings, para, image, list, table)           | `OUTLOOK_IMPORT_PHASE1.md`          |
| 2     | `.eml` upload, Jakarta Mail parsing, inline image extraction                       | `OUTLOOK_IMPORT_PHASE2.md`          |
| 3     | `.msg` upload (POI), banner/footer detection, placeholder heuristics, import report | `OUTLOOK_IMPORT_PHASE3.md`          |
| 4     | LLM-assisted classification for low-confidence blocks (Lovable AI Gateway)         | `OUTLOOK_IMPORT_PHASE4.md`          |

Each phase file contains the concrete code for that phase, so phases can be shipped independently.

## 11. Success Criteria

- Uploaded email renders in the editor with ≥ 80% blocks mapped to typed sections on representative test emails.
- Round-trip: import → save → re-render preview matches the original email visually within acceptable delta.
- No divergence between preview HTML and Outlook HTML — both are still produced by the existing generators from the reconstructed `Section[]`.
- Any block that cannot be mapped survives as an editable `html-content` section (no data loss).
